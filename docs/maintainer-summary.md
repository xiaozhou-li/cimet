# Maintainer Executive Summary: Organizational-Analysis Extension

## What Is Being Added
CIMET is being extended from a primarily technical microservice evolution analyzer into a socio-technical analyzer that adds first-class **organizational structure analysis** derived from version-control contribution history.

The extension introduces a new organizational analysis layer that computes and reports:
- service-level ownership and contributor mapping,
- Pairwise Team Cohesion (PTC) per service and time window,
- organizational coupling between service pairs (OC/NOC) and aggregate coupling (AOC),
- key developer role scores and labels: Jack (breadth), Maven (depth), Connector (brokerage),
- Role Stacking Index (RSI) as co-occurrence intensity across role dimensions,
- longitudinal outputs using configurable temporal windows,
- JSON export compatible with current IR conventions and `example_ir.json` compatibility requirements.

This is an additive scope expansion. Existing technical extraction, delta, merge, detection, and reporting flows remain valid and must continue to run unchanged when organizational mode is disabled.

## Why This Fits the Current CIMET Architecture
The extension is intentionally designed to preserve CIMET’s pipeline-first architecture:
- **Existing pipeline preserved**: config input -> repository acquisition -> IR extraction -> delta extraction -> merge -> detection/reporting/output.
- **Organizational layer inserted, not replacing technical layer**: organizational extraction and metrics consume repo history plus service boundaries generated by existing technical stages.
- **IR/serialization strategy reused**: organizational entities are added using compatible model/JSON patterns rather than introducing an external data path.
- **Detection/reporting style preserved**: metric calculators and reporting modules follow current modular package conventions.

Architecturally, this is a bounded extension: contribution mining, identity normalization, and temporal slicing form a new organizational foundation that feeds metric modules and export/report modules. This minimizes regression risk in the technical core while enabling richer socio-technical analyses.

## Scope and Milestones
The release plan defines six sequential milestones with explicit outputs and quality gates.

### M0: Architecture and Schema Reconnaissance
Outcome:
- Finalize architectural fit, extension points, terminology contract, and IR compatibility mapping.
- Confirm `example_ir.json` compatibility constraints and field-level mapping policy.

Exit signal:
- Clear data contracts and algorithm boundaries agreed; no unresolved naming conflicts in planning package.

### M1: Contribution Extraction and Service Mapping
Outcome:
- Implement commit/file contribution extraction.
- Implement developer identity normalization.
- Map changed files to CIMET service boundaries.
- Build traceable contributor-service relations as organizational base data.

Exit signal:
- Deterministic extraction on fixtures; identity and service assignment rules covered by tests.

### M2: Organizational Cohesion and Coupling Metrics
Outcome:
- Implement ownership concentration and PTC per service-window.
- Implement pairwise OC and normalized pairwise coupling (NOC).
- Implement aggregate coupling (AOC) as window-level mean over active pairs.

Exit signal:
- Metric correctness tests and invariants pass (symmetry, bounded ranges, deterministic reruns).

### M3: Key Developer Roles and RSI
Outcome:
- Implement Jack, Maven, Connector score calculators.
- Implement role classification thresholds and RSI computation.
- Provide per-window role profiles for longitudinal use.

Exit signal:
- Role/RSI algorithms validated on reference scenarios and role-stacking fixtures.

### M4: IR Export and Reporting
Outcome:
- Integrate organizational results into IR-style JSON output.
- Preserve compatibility fields expected by `example_ir.json` while exposing explicit enriched fields.
- Add CLI/config switches and reporting views for organizational analytics.

Exit signal:
- Golden export tests pass against stable fixtures; backward-compatible output behavior verified.

### M5: Hardening, Regression, and Documentation
Outcome:
- Full regression safety for legacy CIMET flows.
- Performance smoke checks and malformed-data resilience.
- Maintainer and user documentation finalized.

Exit signal:
- End-to-end gates green; release documentation complete; known risks explicitly tracked.

## Key Risks Maintainers Should Track
1. **Identity normalization risk**
Contribution analytics are highly sensitive to author alias resolution. Poor normalization can distort ownership, coupling, and role metrics.

2. **Service boundary mapping risk**
Organizational metrics are only meaningful if file-to-service assignment is stable and aligned with CIMET service boundaries.

3. **Temporal sensitivity risk**
Window size/step strongly affects PTC/AOC/roles/RSI trends; defaults and calibration policy must be explicit and versioned.

4. **Metric calibration risk**
Formulas translated from research papers require implementation calibration (edge-case behavior, normalization choices, tie handling).

5. **Compatibility risk for JSON export**
Need to support both explicit enriched fields (e.g., `noc`) and compatibility aliases (e.g., `organizationalCoupling`) without ambiguity.

6. **Performance/scalability risk**
Pairwise service computations and longitudinal windows can increase cost on large histories; bounds and smoke thresholds must be enforced.

7. **Regression risk in technical pipeline**
Any cross-cutting refactor must avoid changing behavior of existing technical modes when organizational analysis is off.

## Recommended Implementation Order
For engineering efficiency and risk control, implement in this order:

1. **Contracts first (M0)**
Freeze terminology, data contracts, and compatibility policy.

2. **Data foundation next (M1)**
Implement extraction, normalization, and service mapping before any advanced metrics.

3. **Core metrics in dependency order (M2)**
Ownership -> PTC -> OC/NOC -> AOC.

4. **Role layer on top of stable metrics (M3)**
Jack/Maven/Connector -> threshold labeling -> RSI -> longitudinal role profiles.

5. **Export/report integration after metric stabilization (M4)**
Lock JSON shape with golden tests before expanding report formats.

6. **Hardening and release readiness last (M5)**
Regression, performance smoke tests, malformed-data handling, and documentation completion.

This sequencing follows the critical path and prevents premature UI/export commitments before metric contracts stabilize.

## What Success Looks Like
Success should be evaluated as both research fidelity and engineering reliability.

### Product/Research success
- CIMET can quantify organizational structure alongside technical evolution.
- Outputs support longitudinal interpretation of ownership, cohesion, coupling, and role dynamics.
- Role dimensions (Jack/Maven/Connector/RSI) are complementary, not collapsed into a single simplistic score.

### Engineering success
- Organizational mode is reproducible and deterministic on fixed inputs.
- JSON artifacts are valid, traceable, and compatibility-preserving.
- Existing CIMET technical behavior is unchanged when organizational mode is off.
- Test gates cover correctness, compatibility, regression, and basic performance limits.

### Maintainer success
- Planning documents form one coherent package with consistent terms and IDs.
- New modules are architecturally bounded and maintainable.
- Open risks and assumptions are explicit, with spike tasks where uncertainty remains.

## Practical Guidance for Maintainers
- Treat `docs/algorithm-spec.md` and data model contracts as implementation ground truth.
- Keep terminology anchored to `docs/planning-glossary.md` to avoid drift.
- Require golden JSON and regression tests for any schema or metric change.
- Version formulas and defaults in outputs for reproducibility.
- Prefer additive extension points over invasive refactors unless required by proven bottlenecks.

With this approach, CIMET can evolve from a technical analyzer into a robust socio-technical platform while preserving the strengths of the current architecture and minimizing operational risk.
